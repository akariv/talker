# Talker — ESP32 Voice AI Intercom

An ESP32-based voice intercom platform. Devices record speech, send it to a
shared AI-powered server (Whisper STT + Claude LLM + OpenAI TTS), and play
back spoken responses. The repo hosts multiple firmware projects that all
talk to the same server and infrastructure.

## How It Works

1. ESP32 boots, connects to WiFi, waits for a button press
2. Records audio via I2S INMP441 mic
3. Streams chunks to server via HTTP POST (`/upload`)
4. Server transcribes with Whisper, sends to Claude with a "say" tool
5. Claude's `say()` calls are queued as responses (possibly to other clients)
6. ESP32 polls `/poll` — server converts text to speech (OpenAI TTS) and returns audio
7. ESP32 plays audio through an I2S speaker, then loops

## Hardware

- **ESP32** (original, no PSRAM required)
- **INMP441** I2S MEMS microphone (SCK=33, WS=25, SD=32)
- **MAX98357A** I2S amplifier + speaker (BCLK=27, LRC=26, DIN=14)
- Button on GPIO 4, LEDs on GPIO 5/18

## Project Structure

```
firmware/
  projects/
    talker/         — full voice intercom firmware
    board-test/     — hardware diagnostic firmware
  components/       — shared ESP-IDF components (WiFi, I2S, HTTP, ...)
server/             — FastAPI voice AI server (Python)
infra/              — Terraform for GCP (Cloud Run, Firestore, IAM, Secret Manager)
scripts/            — provisioning and utility scripts
tests/              — server integration tests
archive/            — retired experiments (gitignored)
```

See `firmware/README.md` for details on building/adding firmware projects.

## Quick Start

### Server

```bash
just install       # install Python dependencies
just server        # run locally on port 8080
```

Requires `ANTHROPIC_API_KEY` and `OPENAI_API_KEY` environment variables. A
test client (`name=test`, `api_key=testkey`) is auto-created in local dev mode.

### Firmware

Requires [ESP-IDF](https://docs.espressif.com/projects/esp-idf/en/stable/esp32/get-started/) v6.0.

```bash
just setup         # one-time: set ESP32 target for the default project (talker)
just provision kitchen   # fetches config from Firestore → writes secrets.h → builds → flashes
```

To work on a different firmware project:

```bash
just PROJECT=board-test setup
just PROJECT=board-test build fm
```

### Deploy to Cloud Run

Push to `main` with changes under `server/**` — GitHub Actions handles it.
Manual trigger:

```bash
just deploy
```

Secrets live in GCP Secret Manager: `anthropic-key`, `openai-key`.

## Configuration

Per-device config (WiFi, server URL, API key) lives in
`firmware/projects/<project>/main/secrets.h`, generated by
`scripts/provision.py` from Firestore. The file is gitignored.

Server-side client registry and shared config live in Firestore:
- `config/wifi`, `config/server`
- `clients/<id>` — one doc per physical device
